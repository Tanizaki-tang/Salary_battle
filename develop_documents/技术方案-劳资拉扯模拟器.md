# 劳资拉扯模拟器技术方案（24H 黑客松精简版）

## 1. 项目目标与范围

### 1.1 目标
- 基于策划案实现一个面向应届生/求职者的谈薪模拟器。
- 当前交付形态优先为在线 `HTML` 网站，前端使用竖屏 UI（移动端比例）进行交互展示。
- 离线 `Electron` 版本列入下一阶段，不纳入本轮 24H 交付范围。
- 核心价值：让用户在多轮谈判中学习薪资谈判技巧、识别 HR 常见话术陷阱，并在复盘中形成知识闭环。

### 1.2 24H 必做范围（Must Have）
- 单场景（只做 1 个岗位场景，避免内容生产爆炸）。
- 初始 Offer 展示（薪资、试用期、五险一金、加班口径）。
- 3~5 回合谈判（文本按钮策略 + 语音输入二选一）。
- 三大指标实时变化（可视化）：
  - `HR满意度/耐心值`
  - `信息暴露度`
  - `陷阱识别计数`
- 终局结算（必须）：`final_score`、最终报价、1 条复盘建议。
- 后端数据库落库（硬性要求）：至少保存 `user_id` 与每局 `final_score`。
- 游客模式即可，但进入游戏时必须生成临时 `user_id` 并落库。
- 必须提供本地 `Mock` 版本：不依赖任何 API，按固定对话树和选项推进回合。

### 1.3 24H 不做范围（Won't Have）
- 多场景/多难度配置后台。
- 完整知识卡系统与复杂法条检索。
- 排行榜、账号体系、社交分享。
- LLM 动态生成话术（先用固定模板+规则）。

## 2. 技术选型

## 2.1 前端（在线 Web 优先）
- `Vue 3 + Vite + TypeScript`
- `Tailwind CSS`
- `Axios`
- 仅保留 3 个页面：开始页、对话页、结果页
- 输出在线静态站点（`index.html + assets`）
- 结构上保留可扩展到 Electron 的能力，但本期不做离线打包

选择理由：
- 快速构建移动端竖屏交互；
- 组件化便于后续扩展语音模式与小程序迁移；
- 与 Python 后端解耦清晰。

## 2.2 后端（Python）
- `FastAPI + Uvicorn`
- 数据库：`SQLite`（在线版本使用，后续可平滑迁移）
- ORM：`SQLAlchemy`
- 数据校验：`Pydantic`
- 不引入 Redis/JWT（黑客松阶段不必要）

选择理由：
- FastAPI 对 JSON API、自动文档、异步支持友好；
- 便于后续接 LLM/语音服务；
- 工程化成熟，部署简单。

## 2.3 对话与语音（精简）
- 对话引擎：固定规则 + 固定 HR 话术模板（不接 LLM）。
- 语音识别：固定使用 `sherpa-onnx`（本地模型文件，离线运行）。
- 语音只做“输入转文字”能力，识别文本沿用同一谈判逻辑。

## 2.4 当前交付通道与后续规划

### 当前（本期）
- 在线 HTML 通道：
  - 前端：`Vue` 打包为静态资源（`index.html + assets`）
  - 后端：`FastAPI` 独立部署（`Nginx + Uvicorn`）
  - 数据：服务端 `SQLite`
  - 语音：服务端 `sherpa-onnx` 推理

### 后续（延期）
- `Electron` 离线通道作为二期扩展，暂不纳入当前里程碑与验收。

## 2.5 双运行模式（必须）
- `api` 模式：前端通过 HTTP 调用 FastAPI（在线环境或本地环境均可）。
- `mock` 模式：前端不发任何网络请求，直接读取本地 `mock` 剧本推进流程。
- 切换方式：通过环境变量 `APP_RUNTIME_MODE=api|mock` 或设置文件控制。
- 目标：在后端/语音模块不可用时，保证 Demo 可完整跑通。

## 3. 系统架构设计

## 3.1 总体架构（24H）
- Web 客户端（在线 HTML）：开始页 -> 对话页 -> 结果页
- FastAPI 服务：开局、回合推进、结算落库、语音识别
- 数据层（SQLite）：持久化用户与对局成绩（服务端）
- AI 语音层（Sherpa-ONNX）：服务端 ASR
- Mock 流程层（Renderer 内）：固定话术 + 固定分支 + 本地状态推进

## 3.4 Local Mock 版本设计（零 API）

### 设计目标
- 不依赖 FastAPI、SQLite、Sherpa-ONNX。
- 前端单独运行即可完成 1 局完整流程（开始 -> 3~5 回合 -> 结算）。

### 实现方式
- 在前端实现 `BattleRuntimeAdapter`：
  - `ApiRuntimeAdapter`：调用真实接口；
  - `MockRuntimeAdapter`：读取本地脚本并返回同构结果。
- 两个适配器返回相同数据结构，页面层无需区分分支逻辑。

### Mock 数据建议
- `mock/scenario.json`：初始 offer、回合上限、初始指标
- `mock/dialog_tree.json`：每回合可选项、HR 回复、指标变化
- `mock/result_rules.json`：最终分数区间与文案

## 3.2 核心模块（按职责强隔离）
- `Game Flow Controller`：唯一编排入口，负责状态机推进、路由分发、结算触发
- `Text Battle Engine`：仅处理文字输入回合，不依赖语音服务
- `Voice Battle Engine`：仅处理语音输入回合，内部调用 `sherpa-onnx` 后复用统一 battle 规则
- `Persistence Layer`：统一落库（`user_id`、`session`、`final_score`）

## 3.3 后端并行开发拆分（你指定的 1+1+1 架构）

### 模块拆分原则（强约束）
- 三个核心模块目录完全独立，互相禁止直接 import。
- 所有模块只能依赖 `contracts`（接口定义）和 `shared_types`（数据结构）。
- 只有 `Game Flow Controller` 可以调用数据库层；battle 模块禁止直接写库。
- 最终集成只改 `orchestrator` 注册表，不改各模块业务代码。

### 成员分工（可并行且不冲突）
- **成员 A：游戏流程控制（1 人）**
  - 负责：`session` 生命周期、回合状态机、模块编排、结算落库触发
  - 不负责：具体文字/语音 battle 规则
- **成员 B：文字 battle（1 人）**
  - 负责：`text-turn` 的策略解析、HR 回复生成、指标变化计算
  - 不负责：数据库、语音识别、路由编排
- **成员 C：语音 battle（1 人）**
  - 负责：`voice-turn` 音频预处理、`sherpa-onnx` 调用、语音回合推进
  - 不负责：数据库、流程状态机

### 统一函数契约（必须遵守）
- `run_text_turn(session_state: dict, text_payload: dict) -> dict`
- `run_voice_turn(session_state: dict, audio_payload: dict) -> dict`
- `advance_game_flow(session_state: dict, turn_result: dict) -> dict`
- `settle_session(session_state: dict) -> dict`
- `save_session_result(user_id: str, session_result: dict) -> dict`

### 集成规则
- `text-turn` 与 `voice-turn` 必须返回同构 `turn_result`，由流程控制模块统一推进状态。
- 流程控制模块只认契约，不关心 text/voice 内部实现细节。
- 任一 battle 模块替换实现时，不影响另外两个模块。

> 只要契约不变，三位成员可独立开发并在最后阶段脚本化合并。

## 4. 数据模型设计（建议）

> 数据库为后端必选项（非可选），用于持久化用户与对局成绩。

## 4.1 关键实体（24H 最小集）
- `user`
  - `id`, `created_at`
- `game_session`
  - `id`, `user_id`, `status`, `round_index`
  - `hr_patience`, `info_exposure`, `trap_count`
  - `final_score`, `created_at`, `ended_at`

## 4.2 核心指标取值
- `hr_patience`: 0~100（越高越好）
- `info_exposure`: 0~100（越低越好）
- `trap_count`: 0~N（越高越好）

## 4.3 最小落库字段（强制）
- `user.id`：用户唯一标识（登录用户或游客临时 ID）。
- `game_session.user_id`：对局归属用户 ID。
- `game_session.final_score`：对局最终得分。
- `game_session.created_at`：对局时间（用于后续统计与排行榜）。

> 即使后续精简功能，也不得移除以上字段的持久化。

## 5. 对话与评分机制（可直接实现）

## 5.1 每回合流程
1. 前端提交用户策略（强势争取 / 迂回试探 / 妥协让步 / 反问施压）。
2. 引擎根据当前状态、场景难度、回合数选取 HR 回复模板。
3. 更新三项指标。
4. 若命中陷阱触发点，计算是否“识破成功”并加分。
5. 返回本回合结果 + 下一回合可选策略。

## 5.2 指标变化（固定参数，现场不调参）
- 强势争取：`hr_patience -8`，`info_exposure -2`
- 迂回试探：`hr_patience -2`，`info_exposure -6`
- 妥协让步：`hr_patience +6`，`info_exposure +8`
- 反问施压：`hr_patience -5`，`info_exposure -5`，若踩中陷阱则 `trap_count +1`

> 注：24H 阶段固定参数，不做大规模调参。

## 5.3 终局判定
- 结束条件：
  - 达到 `max_round`；
  - `hr_patience <= 0`（谈崩）；
  - 用户主动结束。
- 最终报价计算（示例）：
  - `final_salary = base + A*hr_patience - B*info_exposure + C*trap_count`
  - 约束到合理区间 `[scenario_min, scenario_max]`。
- 综合评分（固定）：
  - `score = 0.5*待遇达成 + 0.3*信息保护 + 0.2*陷阱识别`

## 6. API 设计（FastAPI）

## 6.1 路由清单（24H 最小接口）
- `GET /api/v1/health`
  - 出参：服务状态
- `POST /api/v1/sessions`
  - 入参：`user_id`
  - 出参：初始指标、session_id、首句 HR 话术
- `POST /api/v1/sessions/{session_id}/text-turn`
  - 入参：`strategy` 或 `player_text`
  - 出参：`hr_reply`, 指标变化, 是否命中陷阱, 是否结束
- `POST /api/v1/sessions/{session_id}/voice-turn`
  - 入参：`audio_file`
  - 出参：`asr_text`, `hr_reply`, 指标变化, 是否命中陷阱, 是否结束
- `POST /api/v1/sessions/{session_id}/settle`
  - 出参：最终报价、`final_score`（并落库）
- `POST /api/v1/speech/asr`
  - 入参：`audio_file`
  - 出参：`transcript`

## 6.2 响应规范
- 统一格式：
  - `code`（0 成功）
  - `message`
  - `data`
- 错误码示例：
  - `4001` 参数错误
  - `4004` session 不存在
  - `4009` session 已结束
  - `5000` 服务异常

## 7. 前端页面与交互（竖屏）

## 7.1 页面结构（只保留 3 页）
- `/` 开始页（生成游客 `user_id` + 开始）

- `/result/:sessionId` 结果页（分数+一句复盘）

## 7.2 对话页关键交互（简化）
- 上半区：HR 对话气泡流（可滚动）。
- 中间：当前三大指标可视化（进度条 + 颜色提示）。
- 下半区：4 个策略按钮（可在某些回合禁用部分选项）。
- 每次点击后：
  - 按钮短暂 loading；
  - 展示指标涨跌；
  - 返回下一句 HR 回复。

## 7.3 移动端适配要求
- 设计基准宽度：`390px`（iPhone 12 级别）。
- 全站竖屏优先（`min-height: 100vh`）。
- 字体和按钮满足可点击面积（>= 44px）。

## 9. 24H 开发计划（按小时）

## 9.1 时间切片
- `H0-H2`：项目初始化（Vue + FastAPI + SQLite 在线骨架）。
- `H2-H4`：先由流程控制同学定义 `contracts + shared_types + orchestrator` 骨架。
- `H4-H10`：文字 battle 与语音 battle 两位同学并行开发各自模块。
- `H6-H10`：前端 3 页 + 对话主流程打通。
- `H10-H14`：流程控制同学接入两类 battle 模块并打通 `text-turn/voice-turn`。
- `H14-H16`：实现 `mock` 运行时（前端本地数据驱动）。
- `H16-H18`：联合联调与回归（重点测 api/mock 双模式同构结构）。
- `H18-H22`：打包在线 HTML 版本并完成一次外网访问验证。
- `H22-H24`：预演答辩 + 兜底方案（纯文本模式）。

## 9.3 最终组合方式（脚本化集成）
- 入口脚本：`backend/scripts/combine_modules.py`
  - 校验 `flow_controller/text_battle/voice_battle` 是否实现契约函数；
  - 自动注册到 `game_flow_orchestrator.py`（仅更新注册表）；
  - 输出组合结果日志（成功/失败模块清单）。
- 演示脚本：`backend/scripts/run_local_demo.py`
  - 一条命令启动后端并跑通一局样例；
  - 用于答辩前快速验收“接口可用 + 落库成功”。

## 9.2 验收标准
- 用户可完整跑完 1 局对话并拿到复盘结果。
- 三项指标变化符合策略逻辑。
- `sherpa-onnx` 语音输入至少可在安静环境稳定识别短句。
- 数据库中可按 `user_id` 查询到该用户历史对局得分（`final_score`）。
- 在 `APP_RUNTIME_MODE=mock` 下，不启动后端也可完成完整演示流程。

## 10. 测试与质量保障（24H 版）

## 10.1 测试策略
- 冒烟优先：每个接口至少 1 条成功用例。
- 主链路优先：开始 -> 3 回合 -> 结算 -> 落库。
- 端到端优先：手机竖屏可完成完整流程。

## 10.2 必测清单
- `hr_patience` 降到 0 时应立即结算。
- 同一回合不可重复提交策略。
- 结算后 `final_score` 必须成功写入数据库。
- 语音识别失败时可回退文本输入，不阻塞流程。
- Mock 模式断网/无后端进程时，页面流程仍可完整推进。

## 11. 部署方案（黑客松）

## 11.1 环境

## 12. 风险与预案（24H）

- 语音链路不稳定：现场可一键切文本模式继续演示。
- 时间不足：优先保住 start/turn/settle + 落库，复盘文案简化。
- 发布异常：保留开发模式启动脚本（前端 dev + FastAPI）作为兜底。
- 模型文件过大：仅打包必需 `sherpa-onnx` 模型，其他模型按需裁剪。
- 后端临场故障：切换到 `mock` 模式继续演示核心玩法。

## 14. 结论（24H 可交付）

该精简方案以“在线可交付”为本期目标：24 小时内交付在线 HTML 版本（`Vue + FastAPI + SQLite + sherpa-onnx`），并保留 `mock` 兜底模式确保演示稳定。`Electron` 离线版作为下一阶段扩展推进。
